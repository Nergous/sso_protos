# Отчёт об архитектурных улучшениях — `sso_protos`

_Дата: 2026-04-17_
_Ветка: `claude/nightly-suggestions`_

## 1. Краткое резюме

Репозиторий `github.com/Nergous/sso_protos` — компактный пакет gRPC-контрактов
для SSO-сервиса экосистемы `nergous.ru`. Состоит из трёх `.proto`-файлов
(`auth`, `user`, `app`) и сгенерированного Go-кода в `gen/go/sso/`.

### Что сделано хорошо

- **Чёткое разделение по доменам**: `auth`, `user`, `app` — каждый сервис
  в отдельном файле.
- **Консистентный `go_package`** (`nergous.sso.v2;ssov2`) во всех прото-файлах —
  гарантирует один общий Go-пакет.
- **Актуальные версии ключевых зависимостей**: `grpc v1.73.0`,
  `protobuf v1.36.10`, Go 1.23.4 — всё свежее, критических обновлений нет.
- **Документация (`README.md`) содержит** описание RPC, структуру сообщений,
  команды генерации и требования. Удобно для онбординга.
- **Автоматизация генерации через `Taskfile.yaml`** — воспроизводимо.
- **Сгенерированный код закоммичен** — потребители пакета получают рабочий Go
  без необходимости локально запускать `protoc`.

### Что требует внимания

- Отсутствуют CI/линтеры/`buf` — нет защиты от breaking-изменений в контрактах.
- Прото-файлы семантически «сырые»: нет валидаторов, нет пагинации, пароли
  в plaintext-полях, нет кодов ошибок.
- Версия API (`v2`) зашита в `go_package`, но `package auth` не версионирован
  (`auth.v2`) — усложнит будущие миграции.
- Нет тестов (хотя бы smoke-теста на то, что `.proto` компилируются).
- Нет `.gitignore`, нет `LICENSE`, нет `CODEOWNERS`.
- История изменений контракта небезопасна (`int32 -> int64 -> uint32`, смена
  типа поля — breaking change для клиентов).

## 2. Список улучшений

### 🔴 Высокий приоритет

#### H1. Добавить защиту от breaking-изменений (`buf`)
**Проблема:** В истории коммитов видны breaking-изменения типов
(`833de1a int32 -> int64`, `7c1528f int64 -> uint32`). Клиенты, собранные
со старыми контрактами, молча ломаются.
**Шаги:**
1. Добавить `buf.yaml`, `buf.gen.yaml`.
2. Заменить `protoc`-команду в `Taskfile.yaml` на `buf generate`.
3. Настроить `buf lint` и `buf breaking --against '.git#branch=main'`
   в CI.
4. Добавить GitHub Actions workflow `.github/workflows/ci.yml` с шагами
   `buf lint`, `buf breaking`, `go build ./...`, `go vet ./...`.

#### H2. CI-pipeline на GitHub Actions
**Проблема:** Нет автоматической проверки — регрессии попадают в `main`.
**Шаги:**
- Workflow, выполняющий на каждом PR: `go mod tidy -check`, `go build`,
  `go vet`, `buf lint`, `buf breaking`.
- Заблокировать merge без зелёного CI (branch protection rules).

#### H3. Вынести пароли из `RegisterRequest`/`LoginRequest`/`UpdateUserRequest`
**Проблема:** Поле `password` передаётся как `string`. В gRPC-логах, трейсах,
телеметрии оно может оказаться в plaintext. Также `UpdateUserRequest`
смешивает пароль с остальными полями — изменение email случайно обнуляет
пароль, если его забыть передать.
**Шаги:**
1. Пометить чувствительные поля комментарием-предупреждением и гарантировать
   TLS-only транспорт.
2. Для `UpdateUserRequest` перевести поля на `google.protobuf.FieldMask` или
   `optional` (proto3), чтобы отличать «не менять» от «сбросить».
3. Рассмотреть выделение отдельного RPC `ChangePassword(old, new)`.

#### H4. Версионирование proto-пакета
**Проблема:** `package auth;` + `go_package = "...v2;ssov2"` — несогласованно.
Следующая версия (v3) потребует переименования во всех местах клиентов.
**Шаги:**
- Переименовать `package auth;` в `package sso.auth.v2;`.
- Перенести файлы в `proto/sso/auth/v2/*.proto` (buf-style layout).
- Обновить `Taskfile.yaml`/`buf.gen.yaml` и README.

### 🟡 Средний приоритет

#### M1. Добавить пагинацию в `GetAllUsers`, `GetAllApps`, `GetAllUsersForApp`
**Проблема:** `repeated UserModel users = 1;` без `page_size`/`page_token`.
На 10k+ пользователей вызов положит и сервер, и клиент.
**Шаги:**
- Ввести `PageRequest { int32 page_size = 1; string page_token = 2; }`
  и `PageResponse { string next_page_token = 1; }` в соответствии с
  [AIP-158](https://google.aip.dev/158).

#### M2. Единый формат ошибок
**Проблема:** Нет стандарта для возврата ошибок (валидация, not-found,
duplicate). Клиент вынужден парсить строки статуса.
**Шаги:**
- Использовать `google.rpc.Status` с `ErrorInfo`, `BadRequest`.
- Документировать в proto-комментариях коды ошибок для каждого RPC.

#### M3. Валидация входных данных (protoc-gen-validate / protovalidate)
**Проблема:** `email`, `steam_url`, `password` — никак не валидируются на
уровне контракта. Валидация дублируется на каждом сервере/клиенте.
**Шаги:**
- Подключить [`protovalidate`](https://github.com/bufbuild/protovalidate) и
  разметить поля CEL-правилами (`string.email = true`, `string.min_len = 8`
  для пароля, `string.uri = true` для `steam_url`/`link`).

#### M4. Комментарии-документация в `.proto`
**Проблема:** Прото-файлы без единого комментария — а именно они потом
рендерятся в `buf.build` / Swagger.
**Шаги:**
- Добавить doc-комментарии к каждому сервису, RPC, сообщению и полю.
  Это автоматически обогатит README и сгенерирует OpenAPI/документацию.

#### M5. Типы вместо примитивов там, где это осмысленно
**Проблема:** `string path_to_photo` — любая строка; `uint32 user_id` —
можно случайно перепутать с `app_id`.
**Шаги:**
- Для таймстемпов (если появятся `created_at`, `updated_at`) использовать
  `google.protobuf.Timestamp`.
- Рассмотреть обёрточные сообщения `UserId { uint32 value = 1; }`, если
  требуется более строгая типизация (опционально).

#### M6. `google.protobuf.Empty` вместо пустых сообщений
**Проблема:** `LogoutResponse {}`, `UpdateUserResponse {}`,
`DeleteUserResponse {}`, `DeleteAppResponse {}`, `ChangeStatusAppResponse {}`,
`AddAdminResponse {}`, `RemoveAdminResponse {}` — шаблонный шум.
**Шаги:**
- Заменить на `google.protobuf.Empty` или, ещё лучше, спроектировать
  возврат актуального объекта (обновлённый `UserModel`, `AppModel`) — это
  избавит клиентов от дополнительного `GetUser` после `UpdateUser`.

### 🟢 Низкий приоритет

#### L1. Добавить `.gitignore`
Как минимум исключить `*.swp`, `.idea/`, `.vscode/`, `tmp/`.

#### L2. Добавить `LICENSE`
Пакет публикуется для внешнего потребления (`go get`) — без лицензии
формально запрещено использовать.

#### L3. `CODEOWNERS` и шаблон PR
Ускорит ревью и гарантирует внимание мейнтейнеров.

#### L4. Унифицировать именование полей
- `isenabled` → `is_enabled` (snake_case, как остальные поля `is_admin`,
  `path_to_photo`).
- Это breaking-change — планировать вместе с M4/H4.

#### L5. Убрать исполняемый бит с текстовых файлов
`README.md`, `go.mod`, `go.sum`, `*.proto`, `*.pb.go` имеют `chmod +x`
(см. `ls -la`). Для текстовых файлов это некорректно.
```bash
chmod -x proto/sso/*.proto gen/go/sso/*.go go.mod go.sum Taskfile.yaml
```

#### L6. Убрать тестовые коммиты в истории / squash
Коммиты `test`, `test2`, `test3`, `s`, `is it changed?` шумят в истории.
Для будущего — policy на squash-merge в `main`.

#### L7. Таблица совместимости версий в README
Матрица «proto-version × gRPC-version × поддерживаемый сервер» помогает
клиентам подобрать корректную зависимость.

#### L8. Проверка `go mod tidy`
В `go.sum` присутствует запись для `google.golang.org/protobuf v1.36.6`
рядом с `v1.36.10` — вероятно остаточная после обновления. Прогнать
`go mod tidy` и закоммитить чистый lock.

## 3. Рекомендуемый порядок внедрения

1. **Неделя 1:** L1, L2, L5, L8, H2 (инфраструктура и CI).
2. **Неделя 2:** H1 (buf + breaking-check), M4 (документация в proto).
3. **Неделя 3-4:** H3, H4, M2, M3 (семантика контрактов, миграция на v3
   при необходимости).
4. **По мере роста:** M1, M5, M6, L4, L7.

## 4. Метрика успеха

- На каждый PR выполняется `buf breaking` → 0 нежелательных breaking-изменений.
- Каждое `.proto`-поле имеет doc-комментарий (≥ 90 % coverage).
- `go vet ./...` и `buf lint` зелёные в CI.
- README содержит таблицу совместимости и пример использования клиента.
